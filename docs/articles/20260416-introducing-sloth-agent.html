SLOTH AGENT

Introducing a Product-Grade Agent System

Sloth Agent

一个面向中国开发者生态设计的 AI 开发执行系统。它的目标不是再做一个更会聊天的 coding assistant， 而是把 Plan → 开发 → 审查 → 部署 变成一条可控、可恢复、可审计的执行闭环。

今天的大多数 AI 编码体验，在“生成一段代码”时已经足够惊艳；但一旦任务跨越文件、命令、测试、审查与部署， 系统往往开始失去结构。Sloth Agent 的出发点很简单：不要把 agent 当作一次性回答器，而要把它当作持续执行的软件系统。

这也是为什么 Sloth Agent 把运行时内核、工具调用、技能注入、状态持久化、安全默认和质量门控视为同一问题的不同侧面。 真正可用的 agent，不只是“会做”，更要“做得稳、做得清楚、出了问题还能接着做”。

一条执行回路，把阶段 owner、工具调用与状态持久化串在同一个 runtime 里。

Why now

问题不在于模型不够强，而在于执行系统还不够完整

许多 AI coding workflows 已经能写出看起来不错的代码，但它们在长链路任务上仍然容易碎裂：上下文膨胀、工具结果不可追踪、 审查和部署沦为手工补丁、错误恢复依赖“再试一次"。对真正的开发流程来说，这些都不是边角问题，而是系统边界本身。

能生成，不能稳定推进

模型可以提出方案，却缺少一个可靠的执行内核来推进任务状态、处理中断、串联工具调用和质量门控。

能调用工具，不能解释系统

文件读写、命令运行、补丁编辑、测试验证常常是分散的能力，缺少统一的审计视图和恢复语义。

能写代码，不能承担交付责任

没有结构化交接、自动 gate 和回滚路径时，agent 仍然停留在"副驾驶"而非"执行系统"。

Execution Harness

Sloth Agent 先定义一条可靠的执行回路，再谈更大的多 agent 世界

Sloth Agent 的 v1.0 不从“8 个 agent 同时登场"开始，而是刻意收敛到一条更可控的主线： 输入一份已完成的 plan，由 Builder 负责解析、编码、调试和单元测试，再交给 Reviewer 做独立审查，最后由 Deployer 执行部署与验证。

“

真正的 agent 系统，不该只回答“接下来做什么"，还必须能够回答“现在做到哪一步了、为什么停在这里、如何安全地继续下去"。

这也是 Sloth Agent 把运行时、状态、工具和门控视为同一个产品问题的原因。

Infrastructure

让 agent 真正可落地的，不是更多提示词，而是完整的基础设施约束

在 Sloth Agent 里，Tools、Skills、Memory、Context Window、Checkpoints 和 HallucinationGuard 不是彼此独立的卖点。 它们共同构成 agent 的可执行环境：模型能看到什么、能调用什么、调用结果怎样压缩、状态怎样恢复、错误怎样回退，全部需要统一定义。

Tools

以读写文件、运行命令、补丁编辑、搜索为核心能力，通过工具层执行动作并保留结构化结果，确保系统可审计、可回放。

Skills

通过 `SKILL.md` 注入针对任务的高阶行为模板，让模型获得渐进式能力，而不是把所有规则一次性塞进主提示词。

Memory

以文件系统为第一真相源，优先保证简单、可读、可编辑，再向索引与语义层扩展，而不是一开始就把状态埋进黑盒服务。

Context Control

Builder 负责最重的上下文负载，因此需要明确的窗口分区、规则压缩和输出预留，避免长任务在最关键时刻失焦。

Checkpoints

状态持久化不是为了做日志归档，而是为了让 agent 在失败、中断、审批或环境变化后，仍然能够继续同一个 run。

Hallucination Guard

路径白名单、命令黑名单、工具结果验证与运行时边界一起工作，把“模型猜测”尽可能限制在可见、可证、可拒绝的范围内。

Control Surface

它追求的不是更像人，而是更像一个能被团队托付的系统

生产可用性的核心，不是让 agent 看起来“更聪明”，而是让团队能理解、校验、限制并恢复它的行为。Sloth Agent 的很多设计选择， 都是围绕这个控制面展开的。

安全默认

先假设 prompt injection、误操作和越权调用会发生，再通过白名单、黑名单、边界隔离和审计来缩小破坏面。

文件系统即真相

运行状态、检查点、任务产物优先以普通文件存在，让人可以直接查看、修复和理解，而不是依赖不可见状态机。

阶段级模型路由

编码、调试、审查并不一定适合同一个模型。Sloth Agent 直接把“不同阶段用不同模型”视为系统设计，而不是调参细节。

质量门控

审查不是装饰，部署也不是最后补一刀。lint、coverage、smoke test 都被建模为执行流中的硬边界。

成本意识

从上下文窗口到多 provider 路由，系统设计默认考虑 token 与工具成本，而不是把代价隐藏在“更强模型”的背后。

可恢复性

同一个 run 可以因为 gate failure、tool approval、resume 或环境变化而暂停，但不应该因为这些事件被重新拼装成新任务。

Roadmap

这条路线图强调一个非常明确的选择：先把最小的执行闭环做对，再逐步扩成完整的 agent 平台。

Closing

Sloth Agent 想构建的，不是另一种“更会回答”的助手，而是一套更值得托付的软件执行系统。

它从 Builder、Reviewer、Deployer 这条最朴素的链路出发，把运行时内核、工具系统、记忆结构、安全边界和恢复能力压进同一条设计主线里。 如果说下一代 agent 产品的竞争，不再只是模型能力本身，而是谁能把 agent 带入更长、更真实、更可控的工作链路， 那么 Sloth Agent 的答案，就是先把执行系统搭起来。
