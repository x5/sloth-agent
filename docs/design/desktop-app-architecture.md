## Problem Statement

产品经理、项目经理、业务专家、非技术创业者有完整的业务知识和产品想法，但无法独立实现为可运行的功能。他们必须通过开发团队，经历需求沟通、排期等待、反复修改，沟通成本高、周期长、需求理解易偏差。

现有 AI 开发工具（CLI 类 Agent、Cursor、Codex）全部面向程序员，默认使用者懂终端、Git、IDE。非技术用户被使用门槛挡在外面。

## Demand Evidence

**观察到的痛点：** 非技术人员无法直接使用现有 AI Agent 工具。他们需要：
- 不会用终端 / Git / IDE
- 看不懂 Agent 输出的代码和错误
- 无法判断 AI 输出的质量（AI 60 分 vs 需要的 100 分）

**现状替代方案：** 找开发团队实现——沟通损耗、排队等待、需求翻译丢失、反复修改。

**核心洞察：** 产品经理能写出高质量需求文档，但 AI 需要严格的 harness 系统。AI 单独输出只能到 60 分，可提升的 40 分需要工具、流程、质量门控的系统化工作。——这 40 分就是产品价值。

## Target User & Narrowest Wedge

**四类用户（按优先级）：**
1. 产品经理——把需求文档直接变成可运行原型（优先级最高）
2. 业务专家——快速搭建行业业务系统（审批、数据录入、客户管理）
3. 项目经理——自动化项目管理和报告
4. 非技术创业者——快速验证 MVP

**最窄切入：** 让产品经理输入需求文档（自然语言），输出可运行的功能原型，中间通过多 Agent 流水线 + 质量门控保证输出质量。

## Constraints

- 目标用户是**非技术人员**，UI 必须零学习成本
- 需要**实时可视化** Agent 执行过程，让用户看到"它在干什么"
- AI 输出不稳定（60 分→100 分的 gap），需要系统化质量保障
- 需要**流式展示** LLM 响应和工具调用结果
- 现有 Sloth Agent 有大量可复用的 Python 代码（Builder/Reviewer/Deployer/Skill/Cost/Memory/LLM Router/Tool Registry 等 17 个模块）

## Premises

1. AI 单独输出 60 分 + 系统化流程 40 分 = 100 分可运行原型 —— 这个 40 分的产品化是核心竞争力
2. 产品经理不需要学会写代码，但需要能看懂和干预 AI 的输出过程
3. 桌面应用的核心价值不在"能执行代码"，而在于让用户**看到过程、控制质量、理解结果**
4. 先做桌面版（Tauri v2），而非 Web——用户数据隐私、本地文件操作、离线场景都需要桌面形态

## Approaches Considered

### Approach A: 渐进式改造（最小可行）
- 用 Tauri v2 给现有 CLI 包个壳，Python 后端改造成 FastAPI Sidecar
- 复用所有现有 Agent 模块，最小改动量
- UI 只做输入框 + 日志查看器
- 优点：4-8 周可做 MVP，复用率最高
- 缺点：UI 体验粗糙，没有实时流式渲染，本质上是"有 GUI 的终端"
- 不推荐——没有解决非技术用户的核心痛点（看不懂执行过程）

### Approach B: 完整桌面架构（推荐）
- Tauri v2 (React + TypeScript) + Python FastAPI Sidecar + 系统化质量保障
- 前端做完整的 Agent 执行可视化面板、流式输出、干预控制
- 后端用 LiteLLM 做模型抽象，支持远程/本地混合模式
- 将现有 17 个模块重构为独立 Agent 服务
- 优点：解决核心问题，体验完整
- 缺点：3-6 个月工作量，技术复杂度高

### Approach C: 先 Web 验证再桌面
- 纯前端 Next.js + 云端 API，先跑通产品形态
- 优点：2-4 周上线验证，风险最低
- 缺点：无法做本地文件操作，数据隐私问题，无法离线

## Recommended Approach

**Approach B** —— 完整桌面架构。理由：

1. **40 分 gap 需要深度工程化**，不是包个壳能解决的。需要质量门控的可视化、用户干预机制、流式过程展示。
2. **目标用户是非技术人员**，UI 体验是产品的灵魂。粗糙的 UI 等于又做了一次 CLI。
3. **现有代码值得复用但不值得被限制。** 现有 17 个模块可以复用逻辑，但架构需要重写以支持桌面交互。

### 技术栈详细选择

| 层 | 技术选型 | 理由 |
|---|---------|------|
| **桌面壳** | Tauri v2 | 安装包小（10-20MB）、内存低、用户文档已选定 |
| **前端框架** | React + TypeScript | 生态最大，Monaco Editor 支持最好 |
| **代码预览/编辑器** | Monaco Editor（仅用于 Diff/Review 模式） | 默认展示是人类可读的代码摘要。Monaco 只在用户点击"查看代码"时弹出，用于代码对比和 Diff 视图 |
| **UI 组件库** | shadcn/ui + Tailwind CSS | 零运行时依赖，组件可定制，体积小 |
| **状态管理** | Zustand | 轻量，比 Redux 简单 10 倍，足够用 |
| **实时通信** | WebSocket (React → FastAPI) | Agent 执行过程流式推送，通过 WebSocket 直接连接前端和 Python 后端，不经过 Tauri IPC 中间层 |
| **后端** | Python + FastAPI | 轻量、异步、类型提示、与现有代码兼容 |
| **模型路由** | 直连 OpenAI-compatible API（MVP） | MVP 阶段只需 1-2 个 Provider，直接调用即可。LiteLLM 待多 Provider 路由成为真实需求后再引入 |
| **本地存储** | SQLite (via SQLAlchemy) | 会话历史、用户偏好、项目数据 |
| **打包工具** | PyInstaller | 将 Python 打包为二进制 Sidecar |
| **端口管理** | 动态端口检测 + 文件锁 | 避免端口冲突，支持多实例 |

### 核心交互流程

```
用户输入需求 → 需求解析 Agent 生成结构化 Plan
           → Builder 编码 (流式展示进度 + 工具调用)
           → Reviewer 审查 (展示审查结果 + 质量评分)
           → 质量门控 (lint / test / coverage / smoke)
           → 用户确认 (可干预：修改需求 / 接受 / 拒绝重做)
           → Deployer 部署 (可选)
           → 输出可运行原型 (本地文件 + 预览)
```

**用户干预点：** 每一步都展示给非技术用户，用他们能理解的方式：
- 不是显示 "lint passed"，而是显示 "代码格式检查 ✓"
- 不是显示 "coverage: 85%"，而是显示 "测试覆盖了 85% 的代码，还有 15% 没有验证"
- 不是显示 raw log，而是显示"进度条 + 当前步骤 + 预计剩余时间"

### 那 40 分的系统化保障

| AI 60 分的缺陷 | 系统化补足手段 | 用户可见性 |
|----------------|---------------|-----------|
| 代码可能有 bug | 多 Agent 审查 + 自动测试 | 审查报告 + 测试覆盖率可视化 |
| 可能理解错需求 | 需求结构化解析 + 用户确认环节 | 需求理解摘要，用户可修改 |
| 质量参差不齐 | 质量门控（必须过 lint + 测试 + smoke） | 质量门状态面板 |
| 失败不重试 | 自动重规划 + 策略切换 | "正在调整方案，第 N 次尝试" |
| 用户看不懂输出 | 非技术语言翻译 + 代码摘要 | "它做了什么"的自然语言总结 |

### 错误处理与恢复策略

**重试预算：** 每个 Agent 步骤最多重试 3 次。超过后触发"自动重规划"（Adaptive Replanning），调整方案后重新开始。

**降级链：** Builder 失败 → 简化方案重试 → 再失败 → 通知用户并给出建议（"这个需求可能比较复杂，建议拆分成更小的任务"）。

**熔断：** 同一 Provider 连续 3 次错误自动切换到备用 Provider（Phase 1 实现，Phase 0 不需要）。

**用户退出：** 任何时刻用户都可以点击"停止"。停止后保留已生成的代码和进度，可以 Resume。

**Sidecar 崩溃：** FastAPI 进程意外退出，前端检测到连接断开，提示用户"后端服务已断开，正在尝试重启..."，自动重启 Sidecar。

### 性能目标

| 指标 | 目标 |
|------|------|
| 从用户输入到首次可见进度 | < 3 秒（显示"正在分析需求..."） |
| 简单原型生成（单页面应用） | < 5 分钟 |
| 复杂原型生成（多页面+后端） | < 15 分钟 |
| UI 响应延迟 | < 100ms |
| Sidecar 冷启动时间 | < 5 秒 |

### "40 分"如何量化验证

**核心指标：** 定义"原型质量评分"（Prototype Quality Score, PQS），由以下维度组成：
- 需求覆盖率：生成的功能是否覆盖了需求文档中的所有功能点（自动化对比 + 人工确认）
- 可运行性：生成的代码能否直接运行（0 启动错误）
- 代码质量：lint 通过率、测试覆盖率
- 用户满意度：非技术用户对生成结果的主观评分（1-5 分）

**验证里程碑：** Phase 0 结束时，用 5 个真实需求文档测试，PQS 平均值达到 70 分以上才进入 Phase 1。

## Open Questions

1. 多 Provider 并发调用时的成本和并发控制策略——LiteLLM 的 semaphore 具体怎么设
2. 本地模型（Ollama）的打包策略——是引导安装还是内置轻量版
3. 用户数据的备份和同步方案——本地 SQLite 如何跨设备迁移
4. 安装包体积控制——chromadb 等大型依赖的裁剪或替代

## Distribution Plan

**MVP 阶段：只发 Windows**（目标用户主要是国内 PM/业务人员，Windows 占绝对多数）
- Windows: .exe 安装包（NSIS），通过 GitHub Releases 分发
- macOS / Linux：Phase 2 再做

**更新机制：** Tauri 内置 updater，自动检查 GitHub Releases 最新版本，提示用户更新

**CI/CD：** GitHub Actions 自动构建 Windows Release（Phase 2 再扩展到多平台构建）

## Dependencies

- 现有 Sloth Agent 17 个模块的逻辑需要重构适配（不是直接复制粘贴）
- Tauri v2 需要 Rust 工具链（rustup + cargo-tauri）
- Phase 0 不需要 LiteLLM / Ollama / chromadb，先直连 OpenAI-compatible API
- Monaco Editor 通过 npm 引入，前端打包
- PyInstaller 打包 Python 为二进制 Sidecar 有已知痛点（杀毒误报、体积膨胀），需要在 Phase 0 提前验证

## The Assignment

**Phase 0 — 概念验证（2-4 周）：** 不是完整实现 Approach B，而是用最少的代码验证核心假设。

1. **本周：** 画出完整的产品交互原型（Figma），重点展示 Agent 执行过程中非技术用户看到什么、能做什么
2. **第 2 周：** 搭好 Tauri v2 + React 基础框架 + FastAPI Sidecar，跑通"前端输入 → FastAPI 响应 → 流式返回"的最小闭环（用 echo 代替真正的 Agent）
3. **第 3-4 周：** 把现有 Builder Agent 改造成可通过 HTTP 调用的服务，前端能实时看到执行进度。验证"AI 输出 + 质量门控 → 可运行原型"这个核心链路是否真的能跑通

**Phase 1 — 产品化（2-3 个月）：** Phase 0 验证通过后，才开始完整实现 Approach B 的架构。

**Phase 2 — 扩展（3-6 个月）：** 多 Agent 并行、macOS/Linux 支持、LiteLLM 多 Provider 路由、行业业务系统模板

**不急着做的事：** 不要一上来就重写所有 17 个模块。先用一个最简单的 Agent（比如 Skill 查看器）验证桌面壳的通信链路。不要一开始就做 LiteLLM、Ollama、chromadb——这些是 Phase 1 甚至 Phase 2 的事。

## What I noticed about how you think

- 你从"CLI 产品有痛点"出发，直接跳到了技术方案，说明你是**行动派**——想到就做，不等完美方案。这是好信号。
- 你说"AI 60 分 + 系统 40 分 = 100 分"——这是**系统思维**，不是"换个更好的模型"的简单答案。你理解产品价值在流程，不在单点 AI 能力。
- 你坚持用 Tauri 而非 Electron——你有自己的判断，不盲从"更大就是更好"。这是好品味，但也意味着你要接受跨平台调试的成本。
